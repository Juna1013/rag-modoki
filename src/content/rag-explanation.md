# RAG（Retrieval-Augmented Generation）の解説

## RAGとは

RAG（Retrieval-Augmented Generation：検索拡張生成）は、生成AI（LLM）が外部の情報源を検索して取り込み、それを参照しながら回答を生成する仕組みです。

従来のAIはモデル内部の知識に依存するため、「学習後に増えた情報」や「ローカルの資料」を扱うことが苦手でした。RAGはこの問題を、**検索（Retrieval）** と **生成（Generation）** を組み合わせることで解決します。

### RAGを構成する3つの要素

- **Retrieval（検索）** → 保存されたデータから、質問に関連する情報を取り出す
- **Augmented（拡張）** → 検索結果をAIへ追加し、回答生成能力を拡張する
- **Generation（生成）** → 追加された情報を参考に、新しい文章・回答を生成する

## RAGが必要とされる理由

### ① 知識の更新ができない問題

LLMは学習した時点の情報に固定されているため、以下のような "新しい情報" には対応できません：
- 最新ニュース
- 手元のPDF
- 企業内文書

### ② ハルシネーション（事実誤認）が発生

根拠となるデータにアクセスできないと、AIは"推測でそれっぽい回答"を作ってしまいます。

### ③ 情報の根拠が不透明

LLMは「どの情報を元にした回答か」を示すことが苦手です。

## RAGの仕組みと全体像

RAGは大きく分けて **3つのステップ** で動作します：

### STEP 1：知識ベースの準備

外部データ（PDF、テキスト、Webページなど）をAIが扱いやすい形に変換します。

- 文書を小さな単位（チャンク）に分割
- Embedding（埋め込みモデル）でベクトル化
- ベクトルデータベースへ保存

こうして、AIが後で検索できる「意味ベースの辞書」が構築されます。

### STEP 2：質問に関連する情報を検索する

ユーザーの質問も同じくベクトル化し、「意味的に近い」文書を探し出します。

**特徴：**
- キーワード一致ではなく"文意"で検索できる
- 表現が違っていても意味が近ければヒットする
- 大量データから必要な断片だけを正確に抽出できる

### STEP 3：回答を生成する

検索で得た文書をLLMに渡し、「次の情報を参考にして回答を作成してください」と指示します。

これにより、以下が実現されます：
- 根拠のある回答
- 最新情報を反映した回答
- ハルシネーションの抑制

## RAGの技術的ポイント

### ベクトル検索の強み

AIは文章を「数値（ベクトル）」として理解するため、似た意味の文章を柔軟に検索できます。

**例：**
- 「請求書」 ≒ 「支払明細」
- 「事故」 ≒ 「衝突」

単純なキーワード検索より圧倒的に高精度です。

### チャンク分割（Chunking）の重要性

RAGの精度は **文書の分割方法** に大きく左右されます。

- チャンクが大きすぎる → ノイズが増える
- 小さすぎる → 文脈が切れてしまう

一般的には **200〜1000文字** が推奨されます。

### 再ランク（Rerank）

検索結果をさらにAIで評価し、本当に関連する文章だけを取り出すことで精度が上がります。

## RAGのメリット

- **最新の情報を参照できる** → 文書を追加するだけでAIが扱える情報が増える
- **正確性が向上する** → 実際の資料を参照するため、事実に基づく回答が可能
- **透明性が高い** → どの文書を参照したか提示できる
- **再学習が不要で低コスト** → モデルを更新しなくても情報を後付けできる

## RAGの課題・注意点

- 検索が間違うと回答も間違う（Garbage In → Garbage Out）
- チャンク設計に依存する
- 文章を長く渡しすぎるとノイズになる
- ベクトルDBやEmbeddingの品質も重要

## まとめ

RAGは、**検索の正確さ** と **生成AIの柔軟さ** を組み合わせた、現在最も注目されるAI活用手法です。

従来の生成AIが抱えていた以下の課題を解決します：
- 「情報が古い」
- 「根拠が不透明」
- 「事実の誤り」

これにより、最新・正確・根拠のある回答を実現します。

### 活用事例

企業内データやローカルファイルとの連携、専門ドメインの検索システム、FAQボットなど、さまざまな用途で利用されています。
